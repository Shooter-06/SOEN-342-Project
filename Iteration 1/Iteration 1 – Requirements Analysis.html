Iteration 1 – Requirements Analysis

Objective

The purpose of Iteration 1 is to establish the core Requirements Analysis artifacts for the selected critical use case.

Scope of Iteration 1

This iteration focuses only on analysis (not design or implementation). The system is treated as a black box.

Deliverables

1. Use Case Specification

  • Use case name
  • Primary actor
  • Scope
  • Level
  • Preconditions
  • Postconditions (Success Guarantee)
  • Main Success Scenario (numbered steps)
  • At least one Failure Scenario
Critical use cases require detailed specification.

2. Conceptual Class Diagram (Domain Model)

  • Identify conceptual classes (not software classes)
  • Include attributes
  • Include associations with multiplicities
  • Apply generalization when appropriate
  • Respect the “is-a” rule
  • Respect the 100% rule

3. System Sequence Diagram (SSD)

  • Show interaction between actor and system
  • System treated as a black box
  • Include system events
  • Include return values (if any)
  • Include loop notation if repetition exists

4. System Operations

  • Extract operations directly from SSD
  • Operations must match system events
  • Use clear naming conventions (verb-based)

5. Operation Contracts

  • One contract per system operation
  • Include operation signature
  • Include preconditions
  • Include postconditions

Postconditions must describe:

  • Instance creation or deletion
  • Attribute modification
  • Associations formed or broken

Submission Requirements

  • All diagrams must follow UML notation
  • Consistency between use case, SSD, and contracts
  • No design-level details
  • No implementation code

Evaluation Criteria

  • Correct UML notation
  • Conceptual correctness
  • Completeness
  • Internal consistency
  • Application of generalization where justified
